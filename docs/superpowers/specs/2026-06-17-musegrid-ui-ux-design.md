# MuseGrid UI/UX 设计规格

日期：2026-06-17
状态：评审稿
关联文档：`docs/superpowers/specs/2026-06-17-musegrid-product-requirements.md`

## 1. 设计定位

MuseGrid 的视觉方向定义为：

> 未来音乐产业操作系统。

它不是普通后台，不是传统 DAW，也不是泛泛的 AI 生成工具。它应该让用户第一眼感到：这是一个正在重塑音乐产业链的创作基础设施。

设计比例：

- 60% 未来音乐操作系统
- 25% 现代 SaaS 可用性
- 15% 音乐制作质感

核心目标：

- 普通创作者觉得它强大但可上手。
- 幕后创作人觉得它严肃、可信、有资产感。
- 投资人或合作方能感到它不是工具，而是产业链网络。

## 2. 设计关键词

必须体现：

- 未来感
- 科技感
- 创作网络
- 产业链重构
- 分身进化
- 贡献链路
- 音乐生产流程
- 专业可信

必须避免：

- 沉闷灰黑后台
- 普通管理系统模板
- 传统音乐软件的复杂压迫感
- 纯营销官网感
- 过度 AI 紫蓝渐变
- NFT / 加密交易所视觉
- 随机音符装饰
- 玻璃拟态堆叠

## 3. 视觉语言

### 3.1 整体气质

界面应像一个「音乐产业控制中心」，而不是一个表单系统。

具体表达：

- 深色背景有空间感，不是死黑。
- 用发光节点、细线连接、波形、轨道、飞轮表达创作网络。
- 用清晰控件、明确状态和稳定布局保证真实可用。
- 用少量高能量强调色表达生成、播放、成长和贡献。

### 3.2 色彩系统

基础色：

- 背景：近黑石墨、深海军蓝、深炭黑。
- 面板：比背景略亮的暖炭灰或蓝黑。
- 主文字：柔和白，不使用纯白大面积刺眼文本。
- 次文字：冷灰或蓝灰。

强调色：

- 主强调：电青色 / 赛博青绿，用于生成、播放、当前步骤、连接线。
- 辅助强调：酸绿色，用于成长、等级、成功状态。
- 暖强调：琥珀色，用于收益、升级、待审核、提醒。
- 紫蓝色只可作为极少量空间氛围，不作为主按钮和主视觉套路。

禁止：

- 整页紫蓝渐变。
- 大面积单一灰色。
- 随机多彩状态色。
- 低对比灰字。

### 3.3 形态与材质

推荐：

- 8-12px 圆角。
- 细边框。
- 半透明深色层只用于表达空间深度。
- 发光只用于活跃节点、当前流程、播放状态、生成状态。
- 面板阴影要轻，不做厚重卡片堆叠。

禁止：

- 卡片套卡片。
- 所有东西都玻璃拟态。
- 大块发光背景装饰。
- 过度 3D。

### 3.4 字体与排版

中文：

- 使用清晰现代黑体。
- 正文优先可读性。
- 表单、标签、数据必须清楚。

英文：

- MuseGrid 品牌字可更锋利、更科技。
- 英文标签可使用几何无衬线或偏技术感字体。

排版原则：

- 页面标题克制，不做官网式巨型 Hero。
- 工作台内标题要紧凑。
- 数据和金额使用等宽数字或 tabular figures。
- 按钮和状态文字必须完整可读。

## 4. 核心 UX 结构

### 4.1 登录后首页

登录后默认进入平台首页，而不是营销页。

首页目标：

- 让用户快速开始新歌曲项目。
- 展示 MuseGrid 是一个创作网络，而不是单点生成工具。
- 给创作人入口明确但不抢主流程。

推荐模块：

- 新建歌曲项目主入口。
- 最近项目。
- 创作人分身网络可视化。
- 产业循环图：灵感 -> 分身 -> Demo -> 发布 -> 二创 -> 收益 -> 分身成长。
- 成为创作人入口。

视觉表达：

- 中央有明显的「New Song Project」或中文「新建歌曲项目」入口。
- 右侧或下方展示产业飞轮。
- 不做传统三列统计卡堆叠。

### 4.2 创作台

创作台是普通创作者的核心界面。

布局：

- 左侧：项目导航和四步流程。
- 中间：当前环节工作区。
- 右侧：当前创作人分身、贡献链路、额度和生成状态。
- 底部：播放器或生成状态条。

四步流程：

1. 作词
2. 作曲
3. 编曲
4. 制作 / Demo

UX 原则：

- 每一步只有一个主动作。
- 上一步产出自动成为下一步输入。
- 用户可以编辑、重写、确认。
- 当前步骤要有强视觉高亮。
- 未完成步骤不可制造焦虑，应显示清晰等待状态。

未来感表达：

- 四步不是普通 tab，而是创作链路节点。
- 选中的创作人分身以发光节点接入当前步骤。
- 贡献链路随着步骤完成逐步点亮。
- 生成 Demo 时显示波形、能量线或进度轨道。

### 4.3 创作人分身选择器

选择器目标：

- 让用户理解「不同环节有不同创作人分身」。
- 不让用户陷入复杂筛选。
- 强调分身等级、风格和贡献能力。

卡片字段：

- 分身名称
- 创作方向
- Level
- 风格标签
- 代表能力
- 被召唤次数
- 推荐理由

视觉：

- 卡片要像「能力节点」，不是普通商品卡。
- Level 和方向必须一眼可见。
- 选中后该分身应连接到当前创作步骤。

### 4.4 作品结果页

结果页目标：

- 播放生成歌曲。
- 展示歌词和最终 prompt 摘要。
- 显示贡献链路。
- 解释未来二创和收益逻辑。

模块：

- 大号音频播放器和波形。
- 歌词。
- 最终生成 prompt 摘要。
- 贡献链路节点图。
- 模拟 7 天后数据：播放、分享、二创、预计分成。
- 分享按钮。

UX 注意：

- 收益模拟要教育用户理解机制，不要像投资收益承诺。
- 贡献链路要比数据表更直观。
- 播放器是结果页的视觉中心。

### 4.5 创作人入驻

入驻流程目标：

- 让创作人理解自己不是上传简历，而是在创建 Level 1 分身资产。
- 让维护、成长、升级逻辑前置出现。

流程：

1. 选择创作方向。
2. 填写资料和作品案例。
3. 回答风格校准问卷。
4. 预览 Level 1 分身。
5. 提交申请。
6. 进入分身后台。

UX 原则：

- 多步表单必须有进度。
- 字段要解释为什么需要。
- 上传案例和回答问卷要让创作人感到是在训练自己的资产。
- 提交后不要只显示成功，应立即展示分身成长路径。

### 4.6 分身后台

分身后台目标：

- 让创作人看到自己的分身正在成长。
- 强化「真人持续维护」的重要性。
- 展示未来收益想象，但保持可信。

模块：

- 各方向独立等级。
- 中央分身进化核心。
- 维护任务队列。
- 作品案例库。
- 校准问卷。
- 被召唤次数。
- 参与作品。
- 作品播放影响。
- 下一等级要求。
- 预计收益。

视觉表达：

- 可使用分身核心 + 多方向能力模块的结构。
- 等级成长用轨道或时间线，不只是进度条。
- 维护任务要像升级任务，但不要游戏化过头。

## 5. 动效原则

动效必须表达因果，不做纯装饰。

推荐动效：

- 选择分身后，分身节点接入当前步骤。
- 完成一步后，贡献链路节点点亮。
- 生成中显示波形或能量流动。
- 播放时波形轻微运动。
- 分身升级时出现短促、克制的能量点亮。

限制：

- 常规 UI 动效 150-300ms。
- 复杂状态切换不超过 400ms。
- 尊重 reduced motion。
- 不用一直循环的大面积背景动画。
- 不用影响可读性的闪烁。

## 6. 响应式原则

第一版优先桌面 Web。

原因：

- 音乐制作工作流天然更适合桌面。
- 四步流程、分身选择、贡献链路和播放器需要横向空间。

移动端第一版要求：

- 能登录。
- 能查看作品。
- 能播放 Demo。
- 能查看贡献链路简化版。
- 能提交创作人申请的基础信息。

移动端可以暂不完整支持复杂创作台；如支持，应采用纵向步骤流。

### 6.1 跨端设计系统原则

MuseGrid 第一版 Web 必须为后续 App 和小程序沉淀可复用设计系统。视觉效果可以按端适配，但产品结构、组件语义和交互状态要保持一致。

必须沉淀为跨端设计资产：

- 色彩 token：背景、面板、文字、主强调、成长强调、收益强调、错误、成功、警告。
- 字体层级：页面标题、区域标题、正文、标签、数字、按钮。
- 间距体系：4px / 8px 基础栅格。
- 圆角体系：8-12px 为主，特殊节点可更圆但需有规则。
- 状态体系：default、hover、active、selected、disabled、loading、success、error。
- 图标语义：项目、作品、作词、作曲、编曲、制作、分身、贡献链、生成、播放、维护、升级。
- 数据组件语义：贡献链路、分身等级、生成状态、维护任务、作品播放趋势。

跨端应保持一致：

- 四步制作流程的顺序和命名。
- 创作人分身的卡片信息层级。
- 作品结果页的核心信息结构。
- 创作人入驻流程的步骤。
- 分身后台的成长与维护逻辑。

跨端可以适配：

- 桌面 Web 使用左侧导航和多栏工作台。
- App 可使用底部导航和纵向步骤流。
- 小程序优先轻量流程、查看作品、播放 Demo、提交入驻申请。
- 复杂贡献链路在移动端可折叠为分段时间线。

实现提醒：

- 不要只为桌面尺寸设计固定宽度组件。
- 核心组件要能在「横向工作台」和「纵向步骤流」之间切换。
- 动效要有低性能设备降级方案。
- 发光和网络可视化是品牌表达，但不能成为操作理解的唯一方式。

## 7. 组件语言

### 7.1 导航

- 桌面使用左侧导航。
- 当前页面必须清楚高亮。
- 图标加文字，不做纯图标导航。

### 7.2 按钮

- 每个页面只保留一个主按钮。
- 主按钮用于生成、确认、提交、开始项目。
- 次级按钮用于编辑、重写、查看详情。
- 按钮必须有 loading、disabled、error 状态。

### 7.3 表单

- 标签在输入框上方。
- 不用 placeholder 代替 label。
- 错误显示在字段附近。
- 长表单自动保存草稿或明确提示未保存。

### 7.4 数据与图表

推荐图表：

- 节点链路图：贡献链路。
- 飞轮图：产业循环。
- 时间线：分身成长。
- 小型折线图：播放趋势。
- 进度轨道：等级升级。

避免：

- 普通后台大表格占满页面。
- 交易所式 K 线和资产盘。
- 只靠颜色表达状态。

## 8. 关键页面设计要求

### 8.1 平台首页

必须传达：

- 我可以开始做歌。
- 我不是在使用一个单点生成器，而是在进入创作网络。
- 创作人也可以在这里经营分身资产。

### 8.2 创作台

必须传达：

- 一首歌正在按专业流程形成。
- 每一步都可召唤不同方向分身。
- 分身贡献会被记录。
- 最终可以生成可播放 Demo。

### 8.3 作品页

必须传达：

- 结果真实可播放。
- 这首歌有清晰贡献链。
- 未来可以发布、二创、分账。

### 8.4 分身后台

必须传达：

- 分身是可成长资产。
- 真人维护是成长关键。
- 不同创作方向独立升级。
- 未来收益来自调用、作品表现和贡献链路。

## 9. 可访问性与可用性底线

必须满足：

- 正文文字对比度至少 WCAG AA。
- 所有交互目标不小于 44px。
- 键盘可访问。
- 图标按钮必须有文本或 aria-label。
- loading 状态明确。
- 失败状态提供恢复路径。
- 不依赖 hover 才能操作。
- 不因动画影响阅读。

## 10. 实现检查清单

实现时每个主要页面都要检查：

- 是否体现未来音乐产业操作系统，而不是普通后台。
- 是否保留清晰可用的 SaaS 结构。
- 是否有创作网络或贡献链路的视觉表达。
- 是否避免 AI 紫蓝渐变模板感。
- 是否每屏只有一个主动作。
- 是否所有文本在桌面和移动端都不溢出。
- 是否按钮、表单、错误、loading 状态完整。
- 是否移动端至少能查看和播放核心结果。

## 11. 推荐第一版视觉落点

第一版应采用以下组合：

- 平台首页：未来产业飞轮 + 新建项目入口。
- 创作台：四步创作链路 + 分身节点接入 + 底部播放器。
- 作品页：大播放器 + 贡献链路 + 模拟 7 天后数据。
- 分身后台：分身进化核心 + 独立方向等级 + 维护任务队列。

这组设计能同时覆盖需求侧、供给侧和生态叙事，不会把 MuseGrid 做成单纯的 AI 歌曲生成器。
