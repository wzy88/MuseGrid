# MuseGrid v2 产品需求文档

日期：2026-06-25  
状态：产品定义稿  
产品阶段：MVP 到平台化演进需求  
关联原型：`designs/musegrid-music-os-prototype/index-v3.html`

## 1. 一句话定义

MuseGrid 是一个以「创作人分身网络」为核心的 AI 音乐生产平台。普通用户可以从一句灵感开始，按作词、作曲、编曲、制作等环节召唤对应创作人分身协作，生成可播放 Demo；创作人则可以把自己的专业能力沉淀成可召唤、可维护、可成长、未来可收益的分身资产。

MuseGrid 的核心不是「AI 生成音乐」，而是把音乐生产中的专业分工、创作判断、协作贡献和未来收益关系产品化。

## 2. 核心判断

AI 音乐生成已经是拥挤赛道。Suno、Udio、网易天音等产品都在解决「输入提示词，生成歌曲」的问题。MuseGrid 不应成为这些产品的弱化版本。

MuseGrid 的差异化在于：

> 从 prompt 到歌曲，是工具能力；从灵感到一条可追踪、可协作、可成长、可收益的创作链，是平台能力。

因此，MuseGrid 要押注的不是单点生成质量，而是「分身网络」：

- 每个分身代表一个真实或种子创作人的能力资产。
- 每个分身拥有独立方向、等级、风格、代表作、状态和贡献记录。
- 每次召唤都产生可确认的专业交付。
- 每次确认都写入贡献链路。
- 每次作品表现和用户反馈都反哺分身成长。
- 分身主人通过轻量维护让分身持续更像自己。

## 3. 参考观察与竞品边界

### 3.1 LibTV 类 AI 视频工具的启发

用户提供的 AI 视频工具页面有三个值得吸收的结构：

第一，首页不是纯工具页，而是「创作入口 + 最近项目 + 作品广场」的组合。它让用户先看到可开始、可继续、可参考的三类行动。对应到 MuseGrid，首页也不应只放一个 prompt 输入框，而应该同时承载：

- 新建歌曲项目。
- 继续最近创作。
- 浏览作品或分身案例。
- 进入创作人分身网络。

第二，项目工作台使用了「资产画布 + 节点连线 + 多版本结果」的表达方式。AI 视频生产天然有角色图、场景图、分镜图、视频节点等素材关系。MuseGrid 对应的不是画面节点，而是音乐生产节点：

- 灵感。
- 作词结果。
- 作曲方案。
- 编曲方案。
- 制作 prompt。
- Demo 版本。
- 贡献链路。

因此，MuseGrid 的项目页应突出「专家接力链」和「贡献流转」，让用户看到每一步如何成为下一步输入。

第三，AI 视频工具在项目内保留了大量中间资产，而不是只展示最终成片。这个原则对 MuseGrid 非常关键。平台必须保存歌词草稿、Hook 候选、旋律方案、编曲描述、制作 prompt、版本比较和用户确认记录。否则无法形成分身成长、贡献证明和收益分配的基础。

### 3.2 Suno 的启发

Suno 当前核心表达是：用户通过文字、歌词、旋律或音频想法，快速生成包含人声、歌词和制作的完整歌曲。它的优势是低门槛、高速度、强生成质量和清晰的消费级体验。

MuseGrid 应吸收 Suno 的三个优点：

- 启动足够快：普通用户可以从一句话开始。
- 结果足够完整：最终必须能生成可播放 Demo。
- 体验足够轻：不要求用户先理解复杂音乐术语。

但 MuseGrid 不能只复制 Suno 的主路径。Suno 解决的是「我输入想法，系统给我歌」。MuseGrid 要解决的是：

- 我如何召唤不同创作人分身共同完成一首歌。
- 每个分身交付了什么专业判断。
- 我的编辑和确认如何进入作品链路。
- 分身如何因为这次调用而成长。
- 未来作品收益如何有依据地分配。

### 3.3 产品边界结论

MuseGrid 的前台体验要有 Suno 的轻，后台结构要有专业音乐生产的深，中间关系要有分身网络的独特性。

一句话判断：

> Suno 是「生成一首歌」；MuseGrid 是「组建一支可持续进化的线上创作班底」。

## 4. 产品愿景

MuseGrid 的长期愿景是成为「创作人驱动的线上音乐生产网络」。

对于普通创作者：

- 从一句想法开始，快速完成一首可播放 Demo。
- 不需要理解完整音乐工业，也能逐步获得专业协助。
- 在过程中看到每一步谁参与、交付了什么、如何影响下一步。

对于幕后创作人：

- 把自己的作词、作曲、编曲、制作等能力变成分身资产。
- 分身可以被召唤、被采纳、被署名、被持续维护。
- 分身随着调用、作品表现和主人校准逐步成长。
- 未来分身可以带来调用收益、作品收益和二创收益。

对于平台：

- 用创作链路连接用户、分身、作品、贡献、发布和变现。
- 先以工具价值冷启动，再逐步形成分身借用和创作人网络效应。

## 5. 产品定位

MuseGrid 是一个「专家召唤式音乐生产工作台」。

它不是：

- 单纯的 AI 音乐生成器。
- 传统 DAW。
- 普通音乐社区。
- 传统接单平台。
- Web3 / NFT 交易平台。

它是：

- 普通用户完成歌曲 Demo 的创作工作台。
- 创作人创建、维护和运营分身资产的平台。
- 记录作品贡献链路和未来收益分配依据的系统。
- 未来打通创作、制作、推广、发行、变现的音乐产业操作系统。

## 6. 核心产品原则

### 6.1 先召唤专家，再生成音乐

用户不应只看到一个「生成歌曲」按钮。核心体验应该是：

1. 当前要完成哪一个音乐环节。
2. 系统推荐哪些创作人分身。
3. 为什么推荐这个分身。
4. 这个分身会交付什么。
5. 用户是否采纳、修改或要求分身继续改。
6. 确认后如何写入贡献链路。

### 6.2 分身是活资产，不是静态按钮

分身不是「帮我作词」这种功能按钮，而是一个具备人格感、风格边界、代表作、成长状态和维护机制的创作资产。

分身必须让用户感到：

- 它有自己的创作偏好。
- 它有历史作品和代表贡献。
- 它会随着使用和维护进化。
- 它不是一次性训练完就弃置的模型。

### 6.3 贡献链路是平台资产

每一首歌都应该形成贡献链路：

- 哪个环节。
- 召唤了哪个分身。
- 分身当时等级。
- 输出了什么。
- 用户采纳和编辑了什么。
- 确认时间。
- 模拟贡献权重。

第一版可以只做展示和模拟，但数据结构必须为未来署名、收益分配、版权争议、分身成长和二创链路预留。

### 6.4 存证是底层能力，不是前台叙事

平台需要记录作品、分身、交付、确认和协议选择等证据链，但第一阶段不应把「区块链」作为前台主叙事。

用户更关心：

- 这首歌谁参与了。
- 我是否有权发布。
- 分身所有者未来如何分成。
- 如果有争议，平台能否证明创作过程。

前台应使用「贡献证据链」「版权时间戳」「协议确认」等表达。底层未来可以接可信时间戳、司法存证或区块链。

### 6.5 先做工具，再做平台

冷启动阶段不能假设平台已经有大量创作人和分身。第一阶段应允许一个普通用户使用平台提供的种子分身完成完整歌曲。

演进路径：

1. 工具阶段：用户独立使用种子分身做歌。
2. 资产阶段：创作人创建和维护自己的分身。
3. 借用阶段：用户可以调用其他创作人的高质量分身。
4. 网络阶段：分身之间协作产生作品、收益和二创。

## 7. 核心用户

### 7.1 普通创作者

有音乐创作需求，但不具备完整音乐制作能力。

典型场景：

- 给短视频账号做主题曲。
- 给角色、故事、游戏、播客写歌。
- 给生日、婚礼、纪念日、告白场景做歌。
- 有歌词、故事或情绪，但不知道如何变成完整 Demo。
- 是音乐初学者，希望边做边理解歌曲生产流程。

核心需求：

- 快速开始。
- 获得专业协助。
- 生成可播放 Demo。
- 能修改和选择方向。
- 知道每一步谁贡献了什么。
- 最终能分享、发布或继续二创。

### 7.2 创作人

平台供给侧统一称为「创作人」，包括作词人、作曲人、编曲师、制作人、混音师、母带师、推广策划等。

核心需求：

- 创建自己的分身资产。
- 定义分身风格、边界和能力方向。
- 持续维护分身，让它更像自己。
- 查看分身被召唤、采纳、贡献和作品表现。
- 获得未来收益想象和成长路径。

### 7.3 平台运营者

核心需求：

- 管理种子分身。
- 审核真实创作人入驻。
- 监控生成成本和内容风险。
- 观察哪些分身、风格、环节和场景需求最高。
- 管理贡献链路、协议模板和未来结算规则。

## 8. 信息架构

第一阶段建议主导航：

- 创作台：新建项目、继续制作、召唤分身。
- 我的作品：已保存项目、Demo、贡献链路、发布准备。
- 创作人分身网络：浏览、筛选、召唤不同方向分身。
- 申请入驻：创建 Level 1 分身。
- 分身管理：维护分身、查看进化报告、处理校准任务。
- 贡献链路：查看作品贡献、模拟分成、协议记录。
- 账户：额度、会员、协议、个人资料。

未来扩展：

- 推广分身。
- 发行分身。
- 收益中心。
- 版权与协议中心。
- 二创管理。

## 9. 普通创作者主流程

### 9.1 新建项目

用户输入：

- 项目名称。
- 初始灵感。
- 语言。
- 风格。
- 情绪。
- 用途。

可选输入：

- 目标听众。
- 人声倾向。
- 速度偏好。
- 参考方向。
- 使用场景。

系统展示：

- 将为该项目推荐作词、作曲、编曲、制作分身。
- 每一步确认后会形成贡献链路。
- 最终 Demo 生成前才消耗较高额度。

### 9.2 项目制作页：召唤专家工作台

项目制作页是产品核心页面。

布局建议：

- 左侧：歌曲项目摘要和四步专家接力链。
- 中间：当前环节的召唤专家工作台。
- 右侧：当前分身、贡献预写入、下一步输入预览。
- 底部：播放器或生成状态条。

核心交互：

1. 用户进入当前环节。
2. 选择「自己写」或「召唤创作人分身」。
3. 系统展示推荐分身。
4. 用户选择一个分身。
5. 分身接入当前环节。
6. 用户点击召唤。
7. 系统显示分身工作步骤。
8. 分身交付结构化结果。
9. 用户编辑、要求继续修改、换分身对比或确认。
10. 确认后写入贡献链路并解锁下一环节。

### 9.3 作词环节

目标：把粗糙灵感变成歌词方向和歌词初稿。

推荐分身信息：

- 分身名称。
- 创作方向。
- Level。
- 创作格言。
- 风格标签。
- 代表作。
- 被召唤次数。
- 采纳率。
- 最近维护状态。
- 推荐理由。
- 预计交付物。

作词分身交付：

- 主题理解。
- 故事角度。
- Hook 候选。
- 主歌 / 副歌结构。
- 完整歌词草案。
- 押韵、情绪和语言风格说明。
- 分身备注。

用户动作：

- 直接编辑。
- 让分身继续修改。
- 换分身生成对比版本。
- 确认作词成果。

确认后记录：

- 选择的作词分身。
- 输出摘要。
- 用户编辑内容。
- 确认时间。
- 模拟贡献权重。

### 9.4 作曲环节

目标：基于确认歌词生成旋律结构和音乐走向。

作曲分身读取：

- 确认后的歌词。
- Hook 候选和采纳版本。
- 情绪标签。
- 风格标签。
- 速度建议。

作曲分身交付：

- 歌曲结构。
- Hook 情绪。
- 主歌 / 副歌对比。
- 速度范围。
- 和弦或调性方向。
- 旋律描述。
- 给后续编曲使用的输入摘要。

### 9.5 编曲环节

目标：确定乐器配置、节奏律动、段落推进和声音质感。

编曲分身读取：

- 歌词。
- 作曲方案。
- 旋律结构。
- 风格和情绪标签。

编曲分身交付：

- 乐器配置。
- 鼓组和 bass 方向。
- 和声铺底。
- intro / verse / chorus / bridge 推进。
- 音色关键词。
- 给最终制作使用的编曲描述。

### 9.6 制作 / Demo 环节

目标：把前三步确认内容整理成最终生成输入，并调用音乐生成模型生成 Demo。

制作分身读取：

- 最终歌词。
- 作曲方案。
- 编曲方案。
- 项目用途。

制作分身交付：

- 整体制作方向。
- 人声质感。
- 演唱建议。
- 混音倾向。
- 最终风格 prompt。
- 最终歌词包。

Demo 生成逻辑：

- 用户确认最终歌词和 prompt 后才消耗高额度。
- 后端调用音乐生成模型。
- 生成结果持久化到平台存储。
- 生成结果绑定项目、用户、分身贡献链路。

## 10. 分身人格化需求

### 10.1 为什么需要人格化

如果分身只是功能按钮，用户会觉得自己在使用工具；如果分身有名字、偏好、代表作、状态和成长轨迹，用户会觉得自己在组建和培养创作班底。

分身人格化的目标是提升：

- 用户记忆点。
- 情感连接。
- 召唤意愿。
- 创作人资产感。
- 长期留存。

### 10.2 分身基础档案

每个分身需要展示：

- 分身名称。
- 头像或视觉标识。
- 创作方向。
- 当前等级。
- 创作格言。
- 简介。
- 风格标签。
- 擅长场景。
- 不擅长或不接受场景。
- 代表作。
- 示例贡献。
- 被召唤次数。
- 采纳率。
- 维护状态。
- 最近进化记录。

### 10.3 创作格言

创建分身时，系统可根据创作人的作品案例和风格问卷生成 3 条创作格言，由创作人确认或编辑。

示例：

- 「我喜欢先找到情绪转折点，再让一句 Hook 把故事收回来。」
- 「歌词要有画面，但不能把话说满。」
- 「副歌不是喊出来的，是被听众记住的。」

创作格言用于：

- 分身档案页。
- 召唤推荐理由。
- 分身工作过程提示。
- 进化报告中的人格保真判断。

### 10.4 分身状态

分身可以展示轻量状态，用于增强生命感和养成感。

状态示例：

- 状态良好：近期采纳率稳定。
- 需要维护：风格重复度上升。
- 正在探索：新增电子国风方向。
- 风格偏移：连续负面反馈超过阈值。
- 热门召唤：本周期调用显著增加。

状态不应成为随机惩罚，而应来自真实数据或维护任务。

## 11. 创作人分身体系

### 11.1 命名体系

供给侧总称：创作人。

能力方向：

- 作词创作人。
- 作曲创作人。
- 编曲创作人。
- 制作创作人。
- 混音创作人。
- 母带创作人。
- 推广创作人。
- 发行创作人。

数字资产：创作人分身。

### 11.2 独立方向等级

等级按能力方向独立计算，不做统一创作人总等级。

原因：

- 音乐能力天然分工明确。
- 一个优秀作词人不一定擅长制作。
- 独立等级更公平，也更适合推荐和分账。

### 11.3 Level 1 分身创建

新创作人入驻后，不是直接设置价格上架，而是创建 Level 1 分身。

需要填写：

- 创作人名称。
- 创作方向。
- 风格标签。
- 代表作品。
- 创作方法。
- 擅长类型。
- 不擅长或不接受场景。
- 风格校准问卷。
- 平台协议确认。

系统生成初始分身：

- 分身名称。
- 分身简介。
- 创作格言候选。
- 示例贡献。
- 初始风格参数。
- Level 1 状态。
- 成长任务。

### 11.4 分身成长信号

分身成长依据：

- 被召唤次数。
- 参与项目数量。
- 用户采纳率。
- 用户满意度。
- 作品播放数据。
- 完播率。
- 二创数量。
- 分身主人维护质量。
- 人格保真评分。
- 风格稳定度。

## 12. 分身持续进化机制

### 12.1 机制定义

分身不是一次性训练完就弃置的模型，而是通过「定期唤回 + 轻量交互」持续进化的活资产。

核心闭环：

```text
分身被召唤
→ 产生作品
→ 收集作品表现和用户反馈
→ 唤回分身主人做 3 分钟校准
→ 更新风格参数
→ 下一轮召唤更像本人
→ 分身价值变高
→ 更多用户愿意召唤
```

### 12.2 触发机制

常规触发：

- 每 100 次分身被调用，自动触发一次校准邀请。
- 高频分身约 3-5 天触发一次。
- 低频分身可能 2-3 周触发一次。

异常触发：

- 连续 5 首作品评分显著下滑。
- 分身风格单一化严重，例如 90% 作品进入同一模式。
- 用户反馈中负面关键词超过阈值。
- 人格保真评分连续下降。
- 采纳率显著低于同等级平均。

新用户首月触发：

- 创建后第 3 天。
- 第 7 天。
- 第 14 天。
- 第 30 天。

频率控制：

- 强提醒：表现明显下滑、负面反馈超过阈值。
- 弱提醒：累计调用达到周期。
- 静默记录：小波动先不打扰，只进入下次报告。

### 12.3 通知形式

触发后，系统推送到：

- 消息中心。
- App 推送，未来支持。
- 分身管理首页维护队列。

通知文案示例：

> 「林间小调」完成了 100 次新召唤，本周期表现有 2 个方向值得校准。预计耗时 3 分钟。

### 12.4 进化报告

用户点击通知后进入进化报告。

报告分两类指标。

市场表现：

- 本周期生成作品数。
- 平均播放。
- 平均点赞。
- 完播率。
- 分享数。
- 二创数。
- 采纳率。

人格保真：

- 像本人评分。
- 风格重复度。
- 偏离样本。
- 用户负面反馈关键词。
- 分身主人最近维护情况。
- 新风格探索表现。

报告需要展示：

- 表现最好方向。
- 需要关注方向。
- 异常触发原因。
- 预计校准耗时。
- 开始校准入口。

### 12.5 3 分钟校准会话

校准会话分三块。

第一块：本周期表现回顾，约 30 秒。

- 展示关键数据。
- 标记上升和下降项。
- 告知本轮校准目的。

第二块：代表作品打分，约 2 分钟。

系统从本周期作品中挑 4 首代表样本：

- 2 首数据好的。
- 1 首中等的。
- 1 首数据差的。

分身主人回答的问题不是「好不好」，而是：

> 这首歌像不像你本人会写的？

选项：

- 这就是我会写的。
- 有点接近，但不太对。
- 完全不是我。

这个指标是分身精度的核心，不应被播放量完全替代。

第三块：方向性问题，约 1 分钟。

问题示例：

- 最近你的创作兴趣有变化吗？
- 哪个实验方向值得继续？
- 作词习惯最近是否变化？
- 分身目前做到你本人几成水准？
- 接下来希望分身更偏向量产、探索、联合作品还是质量维护？

### 12.6 参数更新摘要

校准结束后，系统展示本轮更新摘要。

示例：

- 古风权重 +0.10。
- 情感叙事权重 -0.05。
- 新增「电子国风」探索标签，初始权重 0.20。
- 降低重复意象「月 / 霜 / 故城」使用频率。
- 触发一次细节校准任务。

用户可执行：

- 确认进化。
- 暂不应用。
- 回滚上一轮。

### 12.7 低成本实现方式

第一阶段不做真正模型重训练，而是将校准结果写入风格参数 JSON。

示例：

```json
{
  "style_weights": {
    "guofeng": 0.72,
    "emotional_story": 0.44,
    "electronic_guofeng": 0.2
  },
  "avoid_patterns": ["过度用典", "重复月霜意象"],
  "preferred_hooks": ["口语化", "画面感", "副歌尾句收束"],
  "persona_fidelity_target": 0.8
}
```

生成时，分身 prompt 和输出约束读取这些参数。

### 12.8 安全与信任

分身主人必须有控制感：

- 每次参数更新可查看。
- 关键更新需要确认。
- 允许回滚上一轮。
- 允许暂停分身被外部召唤。
- 允许删除或隐藏某些样本。

## 13. 贡献链路与收益分配

### 13.1 贡献链路定义

每首作品都必须形成贡献链路。

MVP 记录字段：

- 项目 ID。
- 作品 ID。
- 环节名称。
- 选择的分身。
- 分身所有者。
- 分身方向。
- 当时等级。
- 输出摘要。
- 用户编辑。
- 确认时间。
- 模拟贡献权重。
- 协议版本。

### 13.2 默认贡献权重

第一版建议默认权重：

- 作词分身：20%。
- 作曲分身：25%。
- 编曲分身：25%。
- 制作分身：20%。
- 混音分身：10%。

如果某环节由用户本人完成，该环节权重归项目创建者。

如果未来加入推广和发行分身，其收益更适合从推广收益或发行服务费中计算，不建议直接稀释歌曲创作权重。

### 13.3 收益分配机制

第一阶段：

- 展示模拟分成。
- 不做真实收益结算。
- 不承诺收益。

第二阶段：

- 分身调用可产生平台内收益。
- 高等级分身获得更高推荐权和调用价格。
- 作品播放、打赏、授权收入可按贡献链路生成分配预案。

第三阶段：

- 引入真实结算。
- 支持创作人提现。
- 支持二创收益链路。

### 13.4 贡献链路用途

贡献链路可用于：

- 作品署名。
- 分身推荐。
- 分身等级成长。
- 模拟收益。
- 真实分账。
- 二创链路。
- 版权争议证明。
- 创作人资产履历。

## 14. 协议、版权与存证

### 14.1 核心风险

AI 音乐产品需要从设计之初处理版权问题：

- 用户上传作品是否可用于训练分身。
- 分身基于真实创作人作品生成内容时，授权边界是什么。
- 生成歌曲版权归属如何定义。
- 分身参与贡献是否等于拥有作品版权。
- 平台是否能证明生成过程和用户确认过程。

### 14.2 训练与授权原则

第一阶段原则：

- 用户上传作品只用于训练或校准该用户自己的分身。
- 平台不把用户上传作品混入公共训练池。
- 上传作品前明确授权范围。
- 创作人可以撤回未来训练授权，但已生成作品按当时协议处理。

### 14.3 生成作品权属

建议第一阶段表达：

- 项目创建者拥有生成作品的使用权和发布权，具体以协议为准。
- 平台保留平台内展示、推荐和技术处理权。
- 分身所有者不自动拥有整首歌版权，但拥有贡献记录和未来收益分配依据。
- 如果作品用于商业发行，用户需选择发布协议并确认分成规则。

### 14.4 协议选择

作品生成后，用户发布或导出前需要选择协议。

协议类型示例：

- 内部使用：仅平台内播放和分享。
- 非独家发布：用户可在外部平台发布，平台保留记录和推荐权。
- 独家发行：未来由平台代理发行和结算。
- 商业授权：用于广告、游戏、品牌或影视。

第一版可以只做协议选择界面和记录，不实际接入外部分发。

### 14.5 存证设计

系统自动记录：

- 项目创建时间。
- 每一步分身选择。
- 分身交付内容。
- 用户编辑内容。
- 用户确认时间。
- 最终歌词和 prompt。
- Demo 生成时间。
- 协议选择。
- 贡献权重。

第一阶段前台表达为「贡献证据链」或「版权时间戳」。底层未来可扩展为可信时间戳、司法存证或区块链存证。

## 15. 上下游扩展：推广与发行分身

### 15.1 为什么需要

独立音乐人最大的痛点不只是做歌，而是让歌被听到、被传播、被付费。

MuseGrid 长期要覆盖：

```text
创作 → 制作 → 发布准备 → 发行 → 推广 → 变现 → 二创
```

### 15.2 推广分身

推广分身能力：

- 生成歌曲标题候选。
- 生成专辑封面方向。
- 生成短视频文案。
- 生成小红书 / 抖音 / 微博发布文案。
- 生成宣传切片建议。
- 生成受众定位。

第一阶段实现方式：

- 在作品结果页提供「发布准备」模块。
- 不承诺真实投放。
- 先生成素材包。

### 15.3 发行分身

发行分身能力：

- 生成发行 checklist。
- 检查歌词、封面、音频格式、版权信息是否完整。
- 生成平台上传素材包。
- 未来对接网易云、QQ 音乐、抖音等平台 API。

第一阶段实现方式：

- 做发行准备清单。
- 做导出素材包。
- 不做真实 API 提交。

## 16. 冷启动策略

### 16.1 第一阶段：工具价值

平台先提供种子分身，让用户可以一个人完成五个角色的协作：

- 作词分身。
- 作曲分身。
- 编曲分身。
- 制作分身。
- 可选混音分身。

此阶段目标：

- 用户完成歌曲。
- 用户理解专家接力链。
- 用户理解贡献链路。
- 用户产生第一批作品。

### 16.2 第二阶段：真实创作人入驻

开放创作人入驻，允许他们创建 Level 1 分身。

重点：

- 让创作人理解分身是资产。
- 让创作人看到维护任务和成长路径。
- 先不强调躺赚，强调署名、贡献、成长和未来收益。

### 16.3 第三阶段：分身借用

当平台有足够作品和分身后，开放「借用其他创作人分身」。

此时网络效应开始形成：

- 高质量分身被更多用户召唤。
- 分身获得更多作品表现数据。
- 分身主人更愿意维护。
- 更多创作人愿意加入。

## 17. MVP 范围

### 17.1 必须做

MVP 需要先验证主循环，不追求一次性覆盖完整音乐产业链。优先级按 P0 / P1 / P2 划分。

P0 是没有它产品命题就不成立的能力。P1 是支撑留存、供给侧和平台化的能力。P2 是有利于商业化想象，但不应阻塞第一版上线的能力。

普通用户侧：

- P0：注册 / 登录。
- P0：新建歌曲项目。
- P0：输入灵感、语言、风格、情绪、用途。
- P0：项目制作页。
- P0：作词、作曲、编曲、制作四步流程。
- P0：每一步推荐并选择创作人分身。
- P0：召唤分身生成结构化交付。
- P0：用户编辑、要求修改、确认。
- P0：确认后写入贡献链路。
- P0：调用音乐生成模型生成 Demo。
- P0：作品结果页。
- P0：歌词、最终 prompt、播放器、贡献链路展示。
- P1：模拟收益展示。
- P1：换分身对比生成。
- P1：作品版本历史。

创作人侧：

- P0：平台种子分身可配置和展示。
- P1：申请入驻。
- P1：创建 Level 1 分身。
- P1：填写风格标签、代表作、创作方法。
- P1：初始风格校准问卷。
- P1：分身预览。
- P1：分身管理页。
- P1：维护队列。
- P1：进化报告。
- P1：3 分钟校准会话。
- P1：参数更新摘要。

平台侧：

- P0：种子分身管理。
- P0：贡献链路记录。
- P0：生成成本和调用次数记录。
- P1：创作人申请审核。
- P1：内容风险监控。
- P1：协议选择记录。
- P2：收益中心原型。
- P2：推广 / 发行素材包。

### 17.2 暂不做

- 真实收益结算。
- 创作人提现。
- 真实外部分发 API。
- 完整社区。
- DAW 级音频编辑。
- 分轨混音。
- 多人实时协作。
- 区块链前台功能。
- 真实二创分账。
- 真实外部创作人结算。
- 真实音乐版权登记。

### 17.3 MVP 主循环验收口径

第一版上线前，只看一个核心闭环是否跑通：

```text
用户输入灵感
→ 选择或接受推荐分身
→ 分身在当前环节交付结构化结果
→ 用户编辑或要求修改
→ 用户确认该环节
→ 系统写入贡献链路
→ 下一环节读取上一环节结果
→ 生成 Demo
→ 结果页展示贡献链路
```

如果这个闭环不清楚，即使界面很漂亮，也视为未满足 MVP。

- 轻量露出分身接力链。

- 用户是否理解贡献链路。

### 20.2 供给侧指标

- 创作人入驻转化率。
- Level 1 分身创建完成率。
- 分身维护任务完成率。
- 进化报告打开率。
- 3 分钟校准完成率。
- 参数更新确认率。
- 分身被召唤次数。
- 分身采纳率。
- 人格保真评分。

### 20.3 平台指标

- 成本 / Demo。
- 付费转化率。
- 会员留存。
- 高质量分身数量。
- 分身间协作作品数量。
- 贡献链路完整率。
- 协议确认率。

## 21. 风险与应对

### 21.1 AI 生成同质化

风险：作品听起来都像同一个模型。

应对：

- 强化分身人格化。
- 引入人格保真评分。
- 定期校准分身风格参数。
- 支持换分身对比和继续修改。

### 21.2 创作人信任风险

风险：创作人担心平台只是拿他们训练 AI。

应对：

- 明确上传作品只用于自己的分身。
- 展示贡献记录和未来收益逻辑。
- 提供分身维护、暂停、回滚和删除能力。
- 避免使用剥削感强的产品语言。

### 21.3 版权风险

风险：训练数据、生成作品和商业发布存在版权争议。

应对：

- 明确授权范围。
- 记录贡献证据链。
- 发布前协议确认。
- 商业使用时提供更严格协议流程。

### 21.4 平台冷启动风险

风险：没有足够分身，平台网络无法启动。

应对：

- 第一阶段使用种子分身。
- 先证明工具价值。
- 再开放真实创作人分身借用。

### 21.5 通知打扰风险

风险：频繁唤回创作人维护分身导致反感。

应对：

- 分级提醒。
- 控制频率。
- 每次校准不超过 3 分钟。
- 允许稍后处理和关闭非关键提醒。

## 22. 阶段路线图

### Phase 1：专家召唤工具

目标：普通用户能用种子分身完成歌曲 Demo。

范围：

- 新建项目。
- 四步制作。
- 每步召唤分身。
- 结构化交付。
- 贡献链路。
- Demo 生成。

### Phase 2：分身资产化

目标：创作人能创建和维护自己的分身。

范围：

- 创作人入驻。
- Level 1 分身创建。
- 分身人格化档案。
- 分身管理。
- 维护队列。
- 进化报告。
- 3 分钟校准。

### Phase 3：分身借用网络

目标：用户可调用其他创作人的分身。

范围：

- 分身网络页。
- 推荐算法。
- 调用价格或额度规则。
- 模拟收益。
- 更完整的贡献链路。

### Phase 4：发布与变现

目标：从做歌延伸到发布、推广和收益。

范围：

- 推广分身。

- 协议选择。
- 收益分配预案。
- 外部平台素材包。
- 未来真实结算。

## 23. MVP 验收标准

普通用户侧：

- 用户可以新建项目。
- 用户可以在作词、作曲、编曲、制作四步中选择分身。
- 用户可以召唤分身生成结构化交付。
- 用户可以编辑、要求修改、确认。
- 每次确认生成贡献记录。
- 用户可以生成可播放 Demo。
- 结果页能展示歌词、prompt、贡献链路和模拟收益。

创作人侧：

- 用户可以提交创作人申请。
- 用户可以创建 Level 1 分身。
- 分身有名称、方向、等级、格言、风格标签和代表作。
- 分身管理页展示维护任务。
- 进化报告能展示本周期表现。
- 校准会话能完成样本打分和方向问答。
- 参数更新摘要可确认或回滚。

平台侧：

- 种子分身可用于完整创作流程。
- 贡献链路数据完整。
- 协议选择记录可保存。
- 生成成本和调用次数可追踪。

## 24. 最终产品命题

MuseGrid 要验证的不是「AI 能不能生成音乐」，而是：

需求侧：

> 普通用户是否愿意通过召唤创作人分身，按专业流程完成一首歌。

供给侧：

> 幕后创作人是否愿意把自己的能力沉淀为一个可维护、可成长、未来可收益的分身资产。

平台侧：

> 作品、分身、贡献链路和未来收益机制能否形成正循环。

只要这三个判断成立，MuseGrid 就可以从 AI 音乐工具，继续迭代成创作人网络、作品网络和收益网络。
